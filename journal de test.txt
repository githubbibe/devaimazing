# Fiche racine — run-001

**Nom de la feature** : `todo-list2`

## Objectif

Backend FastAPI minimal exposant deux routes de gestion de tâches, avec stockage en mémoire du processus
(aucune base de données, aucune persistance). Pas de frontend.

## Contexte projet

Repo vierge : aucun `project-map.md`, aucun `architect-map.md`, aucun `templates/` sur disque. Deux 
traces de runs antérieurs existent dans `specs/`, aucun n'ayant dépassé la phase RECEPTION. Le cadrage 
part donc de zéro, sans contraintes d'architecture préexistantes. La structure de cette fiche est 
reconstruite à partir des champs énumérés dans le rôle PM, faute de template disponible.

## Contrat fonctionnel (acté avec l'utilisateur)

### Modèle de tâche

Objet minimal `{id, text}`. Pas de champ `done`, pas de champ de statut, pas d'horodatage.

- `id` : entier, attribué **par le serveur**, auto-incrémenté (1, 2, 3…) à chaque POST accepté.
- `text` : chaîne de caractères, obligatoire, non vide.

### GET /todos

Retourne la **liste courante des tâches en mémoire** : vide au démarrage, et reflétant les tâches créées
par les POST précédents. Ce n'est **pas** un stub figé à `[]` — cette lecture alternative a été 
explicitement écartée par l'utilisateur.

### POST /todos

- Corps attendu : `{"text": "<chaîne non vide>"}`.
- Succès : **201 Created**, corps = la tâche créée avec son `id` attribué par le serveur, soit `{"id": 
1, "text": "..."}`.
- Rejet : **422 Unprocessable Entity** si `text` est absent, vide (`""`), ou d'un type autre que chaîne.
Le rejet se fait au niveau du **schéma Pydantic**, sans logique de validation custom dans le handler.

### Stockage

En mémoire du processus (structure Python simple). **La perte de toutes les données au redémarrage est 
explicitement acceptée par l'utilisateur** — c'est le sens assumé de « en mémoire, pas de base de 
données ».

## Critères d'acceptation (mesurables)

1. `GET /todos` sur un serveur fraîchement démarré → statut `200`, corps `[]`.
2. `POST /todos` avec `{"text": "acheter du pain"}` → statut `201`, corps `{"id": 1, "text": "acheter du
pain"}`.
3. Après N POST valides successifs, les `id` attribués sont exactement `1, 2, …, N`, dans l'ordre des 
requêtes.
4. Après un ou plusieurs POST valides, `GET /todos` → statut `200`, corps contenant les N tâches créées,
chacune sous la forme `{"id": <int>, "text": <str>}`.
5. `POST /todos` avec `{}` (champ `text` absent) → statut `422`.
6. `POST /todos` avec `{"text": ""}` (chaîne vide) → statut `422`.
7. `POST /todos` avec `{"text": 42}` ou `{"text": {"a": 1}}` (type invalide) → statut `422`.
8. Aucun accès à une base de données, aucun fichier de persistance, aucune dépendance de stockage 
externe dans le code produit.

## Périmètre — EXCLU explicitement

- Aucun frontend, aucune page HTML, aucun asset statique.
- Aucune base de données, aucun ORM, aucune migration, aucune persistance disque.
- Aucune route autre que `GET /todos` et `POST /todos` (pas de `GET /todos/{id}`, pas de `PUT`, `PATCH`,
ni `DELETE`).
- Aucun champ `done` / statut / complétion.
- Aucune authentification, aucune autorisation, aucun rate limiting.
- Aucune pagination, aucun filtre, aucun tri paramétrable sur `GET /todos`.
- Aucun déploiement, aucun packaging conteneur, aucune CI.

## Points d'attention pour les agents aval

- **Le rejet de `text: ""` n'est PAS le comportement par défaut de Pydantic** : le défaut accepte la 
chaîne vide. Le critère 6 exige donc une contrainte de schéma explicite (`min_length=1` sur le champ, 
via `Field` ou un type contraint). Cela reste une contrainte déclarative de schéma — et non de la 
logique custom dans le handler — conformément à l'intention de l'utilisateur.

## Questions en suspens — hypothèses à confirmer au checkpoint humain

Les quatre points ci-dessous ont été **proposés par le PM et n'ont reçu aucune réponse explicite de 
l'utilisateur**. Conformément à la règle « un trou d'intention remonte à l'humain, il n'est jamais 
comblé par une valeur par défaut », ils sont inscrits ici comme hypothèses à ratifier, et **non** comme 
critères d'acceptation fermes. Le critère d'acceptation n°3 ci-dessus dépend partiellement de la 
ratification du point 1.

1. **Ordre de `GET /todos`** — hypothèse : ordre d'insertion. Aucun autre ordre n'a été demandé ni 
écarté explicitement.
2. **Doublons de `text`** — hypothèse : autorisés, les `id` étant attribués par le serveur et 
garantissant l'unicité des tâches. Aucune règle de déduplication n'a été demandée.
3. **Longueur maximale de `text`** — hypothèse : aucune borne supérieure (seule la borne inférieure 
`min_length=1` est actée). Aucun plafond n'a été demandé.
4. **Concurrence / multi-worker sur le compteur d'`id`** — hypothèse : hors périmètre, le backend étant 
minimal et supposé mono-processus. L'auto-incrément n'est donc pas protégé contre les accès concurrents 
; en déploiement multi-worker, les `id` collisionneraient entre workers.
